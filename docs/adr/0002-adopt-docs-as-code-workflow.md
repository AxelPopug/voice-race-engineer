# ADR-0002: Adopt docs-as-code workflow

- Статус: Accepted
- Дата: 2026-06-06
- Авторы: команда проекта

## Контекст

Voice Race Engineer развивается через маленькие агентные инкременты. Для такого
процесса недостаточно хранить только код и текущий backlog: важно видеть, почему
решения были приняты, к какому продукту стремится код и какие предположения уже
проверены или еще ожидают проверки.

Документация уже лежит в `docs/`, но до этого не было обязательного правила,
связывающего каждый commit с проверкой влияния на документацию. Без такого
правила агенты могут корректно изменить код, но потерять контекст решения,
устаревший roadmap или новое ограничение.

## Решение

Принять docs-as-code workflow:

- документация хранится рядом с кодом в `docs/`;
- значимые решения фиксируются в `docs/adr/`;
- текущие правила разработки, архитектуры, research и планы остаются living docs;
- каждый commit проходит явный Doc Impact Check;
- если документация не меняется, причина фиксируется в evidence audit или PR body;
- hypotheses, assumptions, unknowns и conflicts должны быть видны на review.

Первый шаг ограничивается Markdown-документацией и процессным gate. MkDocs,
публикация сайта, `docs/index.md` и строгая навигация будут добавлены отдельным
изменением, чтобы не смешивать принятие workflow с выбором tooling.

## Consequences

**+** История решений и направления продукта ревьюится тем же процессом, что и
код.

**+** Agent workflow получает дешевый, но обязательный барьер против
рассинхронизации кода и документации.

**+** Непроверенные предположения остаются допустимыми, но становятся явно
видимыми.

**-** Каждый commit получает небольшой overhead: нужно решить, нужна ли правка
документации, и записать причину.

**-** Пока нет `mkdocs build --strict`, навигация и ссылки проверяются вручную.
Это осознанное промежуточное состояние до отдельного tooling-инкремента.
