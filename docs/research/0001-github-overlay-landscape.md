# Обзор open-source оверлеев и race engineer приложений для iRacing

- Дата исследования: 2026-06-06
- Область: публичные GitHub-репозитории и их документация
- Ограничение: это обзор продукта и заявленной архитектуры, а не полный аудит исходного кода

## Краткий вывод

Рынок обычных iRacing-оверлеев уже насыщен. Standings, relative, radar, track map, inputs, delta и базовый fuel calculator реализованы во многих проектах. Создание еще одного универсального набора виджетов не дает проекту сильного отличия.

Прямой ориентир и главный конкурент идеи — **Crew Chief**. Он уже поддерживает голосовое управление, сообщения о топливе, разрывах, соперниках, стратегии и пит-стопах. Значит, уникальное предложение проекта должно быть уже:

> Voice-first AI race engineer с естественным диалогом, проверяемыми стратегическими расчетами, объяснениями и минимальным overlay.

Нейросеть сама по себе не является преимуществом. Пользовательская ценность появляется, если помощник лучше понимает свободно сформулированные вопросы, учитывает контекст разговора и дает полезный ответ, которого нет среди заранее запрограммированных команд.

## Подробный анализ

### irDashies

Самый интересный референс для архитектуры визуальной части.

Сильные стороны:

- зрелая платформа виджетов, а не один overlay;
- разделение Electron backend, React frontend и общих TypeScript-типов;
- native C++ binding к iRacing SDK;
- Storybook и mock SDK позволяют разрабатывать UI без запущенного iRacing;
- профили, импорт/экспорт layout, OBS/browser source, локальная сеть;
- большое число safety-oriented функций: slow car warning, rejoin indicator, flags.

Что стоит перенять:

- обязательный replay/mock источник данных;
- UI не должен напрямую зависеть от SDK;
- единая типизированная доменная модель между ingestion и presentation;
- отдельный режим разработки без iRacing;
- маленькие окна overlay вместо одного полноэкранного прозрачного окна.

Что не стоит повторять в MVP:

- универсальную платформу с десятками виджетов;
- Electron только ради UI, если voice-first продукт может обойтись минимальным WPF overlay.

### Crew Chief

Главный конкурент и наиболее важный источник продуктовых требований.

Уже реализовано:

- push-to-talk, toggle, always-on и trigger word;
- заранее заданные голосовые команды и варианты фраз;
- запросы топлива, позиции, времени круга, разрывов и состояния машины;
- session/car/full status reports;
- прогноз позиции после пит-стопа;
- подавление некритичных сообщений в поворотах и зонах торможения;
- команды настройки следующего пит-стопа, включая `fuel to the end`;
- отслеживание соперников и событий разных симуляторов.

Сильные стороны:

- зрелая событийная модель race engineer;
- внимание к моменту воспроизведения сообщения, приоритетам и прерываниям;
- большой объем edge cases для разных типов гонок;
- голос является основным интерфейсом, а не дополнением к overlay.

Ограничения и возможность для нашего продукта:

- распознавание основано преимущественно на словаре допустимых фраз;
- пользователь должен знать или угадывать поддерживаемую команду;
- ответы формируются заранее определенной логикой;
- естественный уточняющий диалог и объяснение расчета ограничены;
- вопрос вида «если я следующие пять кругов буду экономить по две десятых, хватит ли мне на еще один круг?» не является естественным базовым сценарием.

Главный урок: нельзя позиционировать проект как «Crew Chief с нейросетью». Нужны измеримые преимущества:

- выше успешность распознавания свободных формулировок;
- контекст нескольких реплик;
- сложные what-if вопросы;
- объяснимый расчет с исходными предположениями;
- строгая проверка чисел перед озвучкой;
- русский язык как полноценный сценарий, если он плохо покрыт конкурентами.

### bo2 Official Overlays / SimHub

Показывает, насколько широка функциональность, которую пользователь уже может получить почти без разработки отдельного приложения.

Уже закрывает:

- standings, relative, spotter, track map и weather;
- fuel consumption, pit lap и fuel to add;
- автоматическую установку количества топлива перед пит-стопом;
- delta, inputs и setup fuel calculator.

Вывод:

- базовый fuel calculator является обязательной функцией, но не отличием;
- автоматические pit commands технически распространены, однако для нашего проекта их лучше исключить из MVP из-за compliance-риска;
- SimHub может использоваться как быстрый инструмент проверки UX-гипотез, но зависимость от него ограничит контроль над voice pipeline и доменной моделью.

### RaceVision, RAH, IR-Overlays и iSimpleRadar

Эти проекты подтверждают несколько устойчивых паттернов:

- overlay почти всегда работает только в windowed/borderless режиме;
- небольшое приложение можно быстро собрать на Python + Qt;
- React/Electron удобен для сложного конфигурируемого UI;
- C#/.NET подходит для легкого Windows overlay и прямой SDK-интеграции;
- большинство небольших проектов останавливаются на визуализации сырых данных.

Их функциональность не конкурирует напрямую с voice-first race engineer, но показывает, что визуальные виджеты легко становятся commodity.

## Архитектурные выводы

### Что строить самостоятельно

- нормализованную доменную модель Race State;
- детерминированный Strategy Engine;
- голосовой orchestrator и типизированные query tools;
- replay-driven тестирование ответов;
- conversational state и what-if calculations;
- минимальный overlay для подтверждения распознанного вопроса, ответа и confidence.

### Что можно изучить или использовать

| Область | Кандидат | Решение |
|---|---|---|
| Паттерн SDK/UI separation | irDashies | Перенять архитектурный подход, не код |
| C# SDK adapter | [IRSDKSharper](https://github.com/mherbold/IRSDKSharper), [iRacingTelemetrySDK](https://github.com/SVappsLAB/iRacingTelemetrySDK) | Провести отдельный code/license benchmark |
| Голосовые сценарии и edge cases | Crew Chief | Использовать как продуктовый benchmark; не копировать код до проверки лицензии |
| Быстрые визуальные прототипы | SimHub | Использовать для UX-экспериментов, не как ядро продукта |
| Overlay UX | irDashies, bo2 | Изучить информационную плотность и приоритеты |

## Рекомендуемое позиционирование

Не строить «еще один overlay». Строить локального голосового стратегического помощника, для которого overlay является вторичным интерфейсом.

Первый демонстрационный сценарий:

1. Пользователь спрашивает: «Хватит ли топлива до финиша?»
2. Помощник отвечает текущим прогнозом и запасом.
3. Пользователь уточняет: «А если проехать на круг больше?»
4. Помощник сохраняет контекст и отвечает требуемой экономией в литрах на круг и процентах.
5. Пользователь спрашивает: «Это реально с моим расходом за последние пять кругов?»
6. Помощник объясняет достижимость и уровень уверенности.

Именно продолжение диалога и what-if анализ отличают продукт от существующих rule-based voice commands и визуальных fuel calculators.

## Изменения к первоначальному ADR

Исследование подтверждает выбор `.NET` для Windows-интеграции и голосового ядра, но снижает приоритет полноценного WPF overlay:

- в MVP overlay должен быть минимальным;
- основной UX — voice-first;
- replay/mock SDK обязателен с первого инкремента;
- универсальные standings/radar/track map не входят в MVP;
- Crew Chief должен использоваться как baseline при оценке качества.

## Метрики конкурентного преимущества

Проект имеет смысл продолжать, если прототип показывает преимущество над словарным voice assistant:

- не менее 90% корректного определения намерения на тестовом наборе свободных фраз;
- 100% озвученных чисел совпадают с результатами Strategy Engine;
- успешный контекстный follow-up без повторения исходного вопроса;
- начало ответа менее чем через 2 секунды для основных запросов;
- отсутствие необходимости запоминать фиксированные команды;
- полезный what-if ответ для экономии топлива, которого нет в базовом Crew Chief сценарии.
